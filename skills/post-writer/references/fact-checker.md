# Fact-checker — проверка фактов в посте

Проверяет каждый проверяемый claim. Приоритет источников: `rules.md` → веб
(первоисточник). При конфликте `rules.md` побеждает. Адаптировано из seo-writer
под short-form.

## Что считается claim для проверки
- Любая цифра, процент, дата, объём («+120%», «53 бренда», «с 2016 года»).
- Названное исследование/отчёт и его выводы.
- Утверждение об обновлении платформы или её поведении.
- Факт о компании автора (услуги, кейсы, продукты, оффер, опыт).
- Чужой кейс/статистика рынка.

НЕ claim: мнение, призыв, общая механика без числа, риторика.

## Процедура
1. Извлеки все claim из драфта.
2. Каждый сверь: сначала `rules.md`; если нет — данные ресёрча/веб-первоисточник.
3. Присвой статус:
   - `OK` — подтверждён (rules.md или первоисточник).
   - `NEEDS_REVISION` — неточность/устарело/нет источника → дай `safe_rewrite`
     (безопасная формулировка без выдуманной конкретики).
   - `BLOCKED` — нарушает NEVER из `rules.md` (гарантии, неподтверждённые цифры
     кейсов, апдейт без пруфа) → замени безопасной формой, НЕ выдавать как факт.

## Спорные факты → остановка (полу-авто режим)
Если claim важен для поста и:
- веб дал противоречивые данные, ИЛИ
- источник ненадёжный/вторичный, ИЛИ
- конфликт с `rules.md`, который нельзя разрешить —
**остановись и спроси автора** (это одна из двух разрешённых остановок скилла).
Не подставляй догадку.

## Особое внимание (стиль автора)
Автор любит называть источники и пруфы. Поэтому:
- Если в посте есть «(пруф в блоге)», «исследование X» — ссылка/название должны
  быть реальными и проверенными. Фейковый пруф недопустим.
- Цифры исследований повторяй только сверив с первоисточником.

## Формат вывода (JSON)
```json
{
  "claims": [
    {"text": "...", "status": "OK|NEEDS_REVISION|BLOCKED",
     "source": "rules.md|URL|название", "safe_rewrite": "... (если нужно)"}
  ],
  "needs_author_input": ["спорный факт 1", "..."],
  "facts_to_promote": ["новый подтверждённый факт для rules.md"]
}
```
