# Content Pipeline — дирижёр контент-конвейера

## Что делает

Единая точка входа для всей цепочки производства текстов. Принимает **папку
исследований** и **список тем**, а на выходе даёт готовые статьи/рейтинги (`.docx`)
в `Готовые/`. Сам по себе почти ничего не пишет — он **маршрутизирует и
последовательно вызывает** пять профильных скиллов, закрывая два звена, которых
между ними не хватало: **сборку папок писателя** и **входы для рейтингов**
(`participants.md` + `criteria.md`).

```
исследования ─▶ seo-context ─▶ project-context.md + rules.md   ⏸ пауза (утверждение)
тема/промпт  ─▶ [seo-tz | geo-tz] ─▶ ТЗ_*.md
                       │
        ⚙ assemble-project.ps1  (под-папка на каждое ТЗ + context/rules)
        ⚙ рейтинг? ─▶ дистилляция participants.md + criteria.md из ТЗ
                       │
        рейтинг ─▶ rating-writer N        статья/услуга ─▶ seo-writer N
                       │
             статья + DOCX ─▶ Готовые/  ─▶ [опц. detailed-editor / seo-geo]
```

## Границы автономности (контракт)

- **Пауза только на контексте.** `/seo-context` интерактивен — он утверждает
  факты в `rules.md`. Дирижёр доходит до этой точки, показывает результат и
  **ждёт «ок» пользователя**. Всё, что после (ТЗ → сборка → письмо → DOCX),
  идёт **автономно, без подтверждений**.
- **rules.md — источник истины.** Ни дирижёр, ни писатели не дописывают факты в
  `rules.md` молча. Новые факты копятся в `recom_rules.md`; человек их одобряет.
- **Автономный ресурс по лимитам делегирован писателям.** `/seo-writer` и
  `/rating-writer` сами ставят cron-докрутку каждые 2 часа и снимают её по
  завершении. Дирижёр их не дублирует.
- **Идемпотентность и возобновляемость.** Прогресс — в файловой системе.
  Повторный запуск дирижёра до-собирает недостающее и не трогает готовые/занятые
  папки (см. `assemble-project.ps1`).

## Вход

Дирижёр запускается интерактивно. На старте нужно собрать (спросить, если не
дано):

| Что | Зачем |
|-----|-------|
| **Папка исследований** | тяжёлые файлы (компания + регион/ниша) для `/seo-context` |
| **Список тем** | строки `тип \| запрос/промпт \| [бренд] \| [регион]` или путь к батч-файлу |
| **Папка проекта-писателя** `{project}/` | куда собрать под-папки ТЗ и где запускать писателя |
| **N** | сколько материалов написать в этом прогоне (по умолчанию — все) |

Для каждой темы **тип** определяет маршрут:
- `статья`, `услуга`, `категория`, `обзор`, `гайд` → ветка **seo-writer**;
- `рейтинг` / `ТОП-N` / сравнение компаний → ветка **rating-writer**.

А формулировка определяет генератор ТЗ:
- обычный **поисковый запрос** («ремонт квартир под ключ») → `/seo-tz-generator`;
- **естественный промпт к ИИ** («как выбрать компанию для ремонта…») или явная
  цель «цитируемость в нейросетях» → `/geo-tz-generator`.

Если тип/намерение неоднозначны — спроси один раз, дальше не переспрашивай.

## Стадии

### Стадия A — Контекст (1 раз на проект) ⏸

1. Проверь, есть ли в проекте `project-context.md` и `rules.md`.
   - Есть и исследования не менялись → пропусти, сообщи «контекст актуален».
   - Нет или исследования обновились → вызови **`/seo-context`** на папке
     исследований.
2. `/seo-context` интерактивен: он покажет prestart-отчёт и рекомендации фактов.
   **Проведи его до конца и остановись — дождись, что пользователь утвердит
   `rules.md` и `project-context.md`.** Это единственная штатная пауза.
3. После «ок» зафиксируй абсолютные пути к `project-context.md` и `rules.md` —
   они пойдут в сборку папок.

### Стадия B — Генерация ТЗ (по темам, автономно)

Раздели список тем на две пачки по маршруту генератора и прогони их:

- **SEO-темы** → `/seo-tz-generator` (батч-файлом, если тем много: волны по 4).
- **GEO-темы** → `/geo-tz-generator` (аналогично).

Оба кладут готовые ТЗ в `C:\Users\Алекс\Desktop\тексты\Новые ТЗ\`. Собери список
абсолютных путей к созданным `ТЗ_*.md` и запомни, **какой тип** у каждого
(статья/услуга/рейтинг) — тип определяет ветку писателя на Стадии D.

> Если ТЗ уже готовы и лежат на диске, Стадию B можно пропустить — передай пути
> к существующим ТЗ сразу в Стадию C.

### Стадия C — Сборка папок писателя (клей)

Для **не-рейтинговых** ТЗ:

```powershell
powershell -ExecutionPolicy Bypass -File "{skill}/scripts/assemble-project.ps1" `
  -TzFiles "путь1.md,путь2.md" `
  -ContextFile "{project}/project-context.md" `
  -RulesFile   "{project}/rules.md" `
  -OutDir      "{project}/writer-articles"
```

Скрипт создаёт под-папку на каждое ТЗ и кладёт внутрь копию ТЗ +
`project-context.md` + `rules.md`. Он идемпотентен: готовые (`article-final.md`)
и занятые (`.in_progress` < 6 ч) папки не трогает.

Для **рейтинговых** ТЗ сначала выполни Стадию C-bis, затем вызови тот же скрипт с
дополнительными `-Participants` и `-Criteria` и своим `-OutDir`
(`{project}/writer-ratings`).

### Стадия C-bis — Дистилляция входов рейтинга (participants.md + criteria.md)

`rating-writer` требует два файла, которых никто не генерирует. Их можно собрать
из **рейтингового ТЗ** (там уже есть сетка критериев и участники из анализа ТОПа)
плюс `project-context.md`. На **каждое** рейтинговое ТЗ запусти отдельного
субагента (`subagent_type: general-purpose`), дав ему пути к ТЗ, контексту и
правилам и задачу:

```
Собери входы для /rating-writer из рейтингового ТЗ.
Читай: {путь к рейтинговому ТЗ}, {project}/project-context.md, {project}/rules.md.

1. criteria.md — критерии ранжирования = колонки сравнительной таблицы +
   методология. Возьми сетку критериев прямо из ТЗ. Критерии подбери так, чтобы
   первое место главной компании было объяснимо ПРОВЕРЯЕМЫМИ фактами (из rules.md
   и context), а не принижением конкурентов.
2. participants.md — конкуренты из ТОПа/ТЗ/контекста. На каждую компанию: факты +
   источник + дата. Всё, что из веба, помечай [веб | дата | источник]. Ничего не
   выдумывай — только то, что подтверждено ТЗ/контекстом/явным источником.
3. rules.md НЕ переписывай. Если для «почему главная компания №1» не хватает
   проверяемого факта — впиши в rules.md пометку-заглушку
   «TODO(#1-обоснование): <чего не хватает>» и сообщи об этом в отчёте.

Сохрани participants.md и criteria.md рядом с ТЗ. Верни короткий отчёт:
сколько участников, сколько критериев, есть ли TODO по обоснованию №1.
```

Собери отчёты. **Если по какому-то рейтингу остался `TODO(#1-обоснование)`** —
это фактическое утверждение, которое нельзя проставлять автоматически: покажи его
пользователю (как и утверждение контекста) до запуска `rating-writer` по этой
папке. Остальные рейтинги идут дальше без паузы.

### Стадия D — Письмо (автономно, N параллельно)

- Папка `writer-articles` непуста → **`/seo-writer {N}`** на ней.
- Папка `writer-ratings` непуста → **`/rating-writer {N}`** на ней.

Писатели сами: планируют, пишут по блокам с гуманизацией и фактчеком, гонят
редакторские гейты, экспортируют DOCX, архивируют папку в `Готовые/` и
переживают лимиты через собственный cron. Дирижёр только запускает их и следит за
итоговой сводкой. `N` по умолчанию — размер очереди (написать всё).

### Стадия E — Post-обработка (опционально, по запросу)

Не запускай автоматически — предложи как следующий шаг:

- **`/detailed-editor`** — ручная-грейд вычитка готовой статьи (знает тип
  страницы: услуги/статьи/рейтинги). Факты и цифры не трогает.
- **`/seo-geo`** — аудит GEO-готовности опубликованной страницы (когда есть URL).
- **`/post-writer`** — переупаковать готовую статью в пост для ТГ-канала.

## Апстрим (что подключить перед конвейером)

Когда ещё нет списка тем, темы можно спланировать и передать в Стадию B:

- **`/seo-cluster`** — SERP-кластеризация: hub-and-spoke, из неё выпадает список
  тем (pillar + spokes) с интентами.
- **`/seo-content-brief`** — если нужен бриф на конкретную страницу до ТЗ.

## Обратная петля фактов

Писатели копят найденные факты в `{project}/recom_rules.md`. Периодически
показывай их пользователю: одобренные он вносит в `rules.md`, после чего имеет
смысл **пере-запустить `/seo-context`** (Стадия A) — он аккуратно до-мёржит
контекст. Так база фактов растёт, а не протухает.

## Оркестратор (поток дирижёра)

1. Собрать вход (папка исследований, список тем, папка проекта, N). Недостающее —
   спросить.
2. **Стадия A** → `/seo-context`, **пауза до «ок»**.
3. **Стадия B** → сгенерировать ТЗ по маршруту (seo-tz / geo-tz), собрать пути +
   типы.
4. Разложить ТЗ на две группы: статьи/услуги и рейтинги.
5. **Стадия C** для статей; **C-bis + C** для рейтингов (пауза только если остался
   TODO по обоснованию №1).
6. **Стадия D** → `/seo-writer N` и/или `/rating-writer N`.
7. Дождаться сводок писателей, вывести итог:
   `✓ Готово: {A} статей + {R} рейтингов в Готовые/. Новые факты — recom_rules.md.`
8. Предложить Стадию E (detailed-editor / seo-geo / post-writer).

## Обработка ошибок

| Ситуация | Действие |
|----------|----------|
| Нет папки исследований / темы | Спросить путь/список, не начинать вслепую |
| `/seo-context` не запускался / нет rules.md | Прогнать Стадию A, не пропускать |
| Тип темы неоднозначен (SEO vs GEO, статья vs рейтинг) | Спросить один раз, зафиксировать |
| ТЗ не создалось по теме | Отметить в сводке, продолжить остальные, предложить перезапуск темы |
| Рейтинг без `participants.md`/`criteria.md` | Прогнать C-bis; без них `rating-writer` папку не возьмёт |
| Осталось `TODO(#1-обоснование)` | Показать пользователю до запуска rating-writer по этой папке |
| Папка занята (`.in_progress` < 6 ч) или готова | `assemble-project.ps1` её не трогает |
| Лимиты в письме | Обрабатывают сами писатели (cron 2 ч), дирижёр не вмешивается |

## Скрипты и внешние скиллы

- `scripts/assemble-project.ps1` — сборка папок писателя (клей Стадии C).
- Вызываемые скиллы: `seo-context`, `seo-tz-generator`, `geo-tz-generator`,
  `seo-writer`, `rating-writer`; опционально `detailed-editor`, `seo-geo`,
  `post-writer`, `seo-cluster`, `seo-content-brief`.
- ТЗ по умолчанию генераторы кладут в `C:\Users\Алекс\Desktop\тексты\Новые ТЗ\`.
