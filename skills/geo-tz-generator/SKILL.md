# GEO ТЗ Генератор (ТЗ под продвижение в нейросетях)

Скилл создаёт ТЗ для копирайтеров, цель которых — **цитируемость контента в
поисковых ИИ** (Google AI Overviews в первую очередь; ChatGPT Search, Perplexity,
Gemini, Яндекс Нейро — вторично, механика общая), а не классическое ранжирование
в десятке. Сосед `seo-tz-generator` затачивает страницу под SERP; этот скилл —
под то, как ИИ **разбирает запрос на веер подзапросов и пересобирает ответ из
цитат**.

Ключевой факт, на котором стоит вся методология: страницы, релевантные и основному
запросу, **и** его fan-out-подзапросам, на **161% чаще** попадают в цитаты AI
Overviews; контент, релевантный только fan-out-подзапросам, на **49% чаще**
цитируется, чем релевантный только головному запросу. Вывод для ТЗ: статья обязана
покрыть **весь веер**, а не один запрос, и подавать каждый ответ
**извлекаемым самодостаточным чанком**.

Пайплайн из трёх фаз:

```
Фаза 1  Fan-Out    промпт → веер подзапросов (6 типов) + слоты      [модель]
Фаза 2  Citations  WebSearch по подзапросам → citation-map.json     [модель+WebSearch]
Фаза 3  Synthesize матрица покрытия + extractability → сборка ТЗ     [модель]
```

> **Почему нет Node-скрипта, как в `seo-tz-generator`.** Там «жёсткую» фактуру даёт
> внешний HTTP-API (XMLRiver), недоступный модели напрямую, поэтому нужен Node.
> Здесь источник фактуры — `WebSearch`, а это **инструмент модели**. Фазу 2 ведёт
> сама модель и фиксирует результат в `out/citation-map.json` (через Write) —
> принцип «ссылайся на зафиксированную фактуру» сохраняется, скрипт не нужен.

**Два режима работы:**
- **Одиночный** — одна тема строкой `тип | промпт | бренд | регион`. Проходишь
  Фазы 1–3 сам и сохраняешь одно ТЗ. Описан ниже по фазам.
- **Батч** — файл со списком тем. Каждое ТЗ пишет отдельный субагент в
  изолированном контексте, параллельно до 4. См. [«Батч-режим»](#батч-режим).

## Формат ввода (одиночный режим)

```
тип | промпт | бренд | регион
```

- **тип** — тип страницы (статья / услуга / рейтинг / обзор / гайд…). Влияет на
  schema и структуру (рейтинг → сравнительная таблица + критерии). Это всё равно
  **ТЗ копирайтеру**, а не сама статья.
- **промпт** — формулировка, которую целевой пользователь задаёт нейросети.
  Не ключевое слово, а **естественный вопрос**: «как выбрать компанию для ремонта
  загородного дома под ключ», «что лучше — дизайн и ремонт в одной компании или
  раздельно».
- **бренд** — продвигаемая компания/продукт (для citation-bait, перелинковки,
  Gap-анализа). Необязателен — тогда ТЗ без брендовой привязки.
- **регион** — необязателен; по умолчанию Москва.

Примеры:
- `статья | как выбрать компанию для ремонта загородного дома под ключ | АртСтрой | Москва`
- `услуга | сколько стоит дизайн-проект квартиры | Che Design`
- `рейтинг | лучшие студии дизайна интерьера в Москве | | Москва`

## Фаза 1 — Fan-Out Decomposition (разложение промпта на веер)

Цель — превратить один промпт в **список конкретных подзапросов**, которые ИИ
сгенерирует и разошлёт по источникам, плюс **слоты**, которые он попытается
заполнить. Это смысловой реверс-инжиниринг, опирающийся на
[references/fan-out-taxonomy.md](references/fan-out-taxonomy.md) — прочитай её
перед началом.

### Шаг 1.1 — Разбор промпта
- **NER / сущности и атрибуты:** какие сущности в промпте (тип, нормальная форма),
  явные и неявные атрибуты.
- **Intent-классификация:** информационный / коммерческий / транзакционный /
  навигационный (часто смешанный — перечисли по приоритету).
- **Слоты (implicit fan-out):** переменные, которые ИИ попытается заполнить —
  бюджет, сроки, регион, тип объекта, размер, гарантии, риски. Каждый
  незаполненный слот = отдельный подзапрос.

### Шаг 1.2 — Генерация веера (6 типов)
По таксономии разложи промпт минимум на **12–20 подзапросов**, распределённых по
типам (норма — каждый тип представлен):

| Тип | Что покрывает |
|-----|---------------|
| **Related** | смежные аспекты той же темы |
| **Comparative** | «X vs Y», выбор между вариантами |
| **Technical / specific** | характеристики, этапы, реализация, цифры |
| **Contextual** | гео / тип объекта / сценарий / сегмент аудитории |
| **Temporal** | свежесть: «в 2026», новые требования, актуальные цены |
| **Implicit (slots)** | подзапросы из незаполненных слотов Шага 1.1 |

Выход Фазы 1 — **таблица веера**: `подзапрос · тип · приоритет (H/M/L) · слот`.
Приоритет ставь по близости к интенту промпта; финально его подтвердит Фаза 2.

## Фаза 2 — Citation Landscape (кто и как уже цитируется)

Опора — `WebSearch`. Цель — собрать **фактуру о цитируемости**: какие источники
всплывают по вееру, в каком формате подан цитируемый ответ, какие подвопросы
повторяются. Фокус-движок — **Google AI Overviews** (для РФ-ниши держи в уме
Яндекс Нейро — механика fan-out та же).

### Шаг 2.1 — Прогон
Прогони через `WebSearch`:
1. сам **промпт** (как есть, естественной формулировкой);
2. **топ-подзапросы** Фазы 1 с приоритетом H и частью M (8–12 запросов).

Для РФ-тематики добавляй регион в формулировку запроса (выдача WebSearch — US;
региональные/языковые сигналы компенсируй словами в самом запросе: «в Москве»,
«в России 2026»).

### Шаг 2.2 — Сбор citation-map.json
Зафиксируй наблюдения в `out/citation-map.json` (создай через Write):

```json
{
  "prompt": "...",
  "region": "Москва",
  "engine_focus": "google-ai-overviews",
  "subqueries_checked": [{ "q": "...", "type": "comparative", "priority": "H" }],
  "sources": [
    { "domain": "...", "url": "...", "cited_for": ["подзапрос"],
      "format": "definition|list|table|number|faq|howto",
      "passage_example": "коротко, как подан цитируемый ответ" }
  ],
  "recurring_subquestions": ["вопрос, всплывающий у многих источников"],
  "formats_seen": { "table": 4, "faq": 6, "number": 9 },
  "gaps": ["подзапрос Фазы 1, который никто не раскрывает"]
}
```

**Правила фиксации:**
- `format` — в каком виде подан цитируемый кусок (определение / список / таблица /
  голая цифра / FAQ / пошагово). Это потом диктует требования к чанкам в ТЗ.
- `recurring_subquestions` — подвопросы, повторяющиеся у разных источников: это
  **реальный fan-out ниши**, поднимай их приоритет в матрице покрытия.
- `gaps` — подзапросы Фазы 1, которые **никто не раскрывает**. Это точки роста
  (Content Effort, см. методологию).

**Прочитай свой `out/citation-map.json` перед Фазой 3** — это фактура ТЗ.

## Фаза 3 — Synthesize (сборка ТЗ под цитируемость)

Внедри все требования из
[references/geo-methodology.md](references/geo-methodology.md). Везде, где даёшь
приоритет или цифру, **ссылайся на `citation-map.json`** («формат — таблица, 4/N
источников подают так», «подзапрос в `recurring_subquestions` → Mandatory»).

### Шаг 3.1 — Матрица покрытия fan-out (ядро ТЗ)
Сведи веер Фазы 1 и фактуру Фазы 2 в таблицу. **Каждый подзапрос → отдельный
самодостаточный H2/H3-чанк.** Приоритет:
- **Mandatory** — подзапрос в `recurring_subquestions` ИЛИ цитируется источниками
  Фазы 2;
- **Recommended** — H/M-подзапрос Фазы 1 без подтверждения цитатой;
- **Gap (обязательный)** — из `gaps`: никто не раскрыл → уникальное усилие.

| Подзапрос | Тип fan-out | Приоритет | Формат чанка | Обоснование (citation-map) |
|-----------|-------------|-----------|--------------|----------------------------|

### Шаг 3.2 — Extractability (извлекаемость каждого чанка)
По каждому Mandatory-чанку задай в ТЗ:
- **вопросный H2/H3** (формулировкой подзапроса, как его задаёт пользователь);
- **прямой ответ в первом 1–2 предложении** блока (определение / число / вывод),
  до любой воды — чтобы ИИ мог вырезать чанк целиком;
- **формат** из `formats_seen` (если ниша отвечает таблицей/списком — требуй их).

### Шаг 3.3 — Schema (JSON-LD)
Минимум **Article + FAQPage**; по типу страницы добавь HowTo (гайд/инструкция),
Product/Review/AggregateRating (услуга/рейтинг). FAQPage строй из подзапросов
веера — это машинно-читаемый fan-out.

### Шаг 3.4 — Citation-bait и бренд
- явные **цифры, определения, сроки** как «крючки» для цитаты (ИИ охотнее
  цитирует конкретику, чем общие слова);
- если задан **бренд** — заложи 1–2 уникальных факта/кейса/расчёта от бренда,
  которых нет в `sources` (закрывает `gaps` и даёт E-E-A-T).

### Шаг 3.5 — Topical authority и перелинковка
Обозначь pillar-страницу и cluster-подзапросы, которые стоит вынести в отдельные
материалы (веер шире одной статьи). Заложи внутренние связи.

### Методология (обязательно)
Внедри 4 правила из [references/geo-methodology.md](references/geo-methodology.md):
Content Effort (через `gaps`), декларативные заголовки, Original Media,
сигнальная когерентность Title ↔ H1 ↔ первый абзац.

## Формат итогового ТЗ (шаблон)

1. **Метаданные:** Title, Description — с опорой на промпт и `recurring_subquestions`.
2. **Целевой промпт и веер:** исходный промпт + таблица fan-out (Фаза 1).
3. **Матрица покрытия:** таблица из Шага 3.1 (подзапрос → чанк → приоритет).
4. **Структура (план):** H1 → H2 → H3, у каждого H2 — тип fan-out, приоритет,
   требуемый формат чанка и прямой ответ-крючок.
5. **Gap-блоки:** подзапросы из `gaps` → обязательное уникальное усилие.
6. **Extractability-чек-лист:** вопросные заголовки, ответ в первом предложении,
   форматы (таблицы/списки/FAQ), длина чанков.
7. **Schema:** перечень типов JSON-LD + из чего собрать FAQPage.
8. **Ключевые сущности и термины:** сущности Фазы 1 + лексика цитируемых пассажей.
9. **Фактура и citation-bait:** цифры/факты/кейсы (в т.ч. брендовые) для цитат.

### Куда сохранять готовое ТЗ

**Папка по умолчанию:** `C:\Users\Алекс\Desktop\тексты\Новые ТЗ\`
Имя файла — `ТЗ_GEO_<Короткое_название>_<Регион>.md` (пробелы → подчёркивания).
Если файл с таким именем уже есть — добавь дату `_ГГГГ-ММ-ДД`.

Промежуточный `out/citation-map.json` остаётся в `out/` — в папку с готовыми ТЗ
кладётся **только финальный Markdown**.

## Батч-режим

**Триггер:** пользователь скинул файл (`.md`/`.txt`) со списком тем или попросил
сделать ТЗ «пачкой / по списку / из файла». Тогда ты — **оркестратор**: парсишь
файл и раздаёшь темы субагентам, по одному ТЗ на субагента, в изолированных
контекстах (тяжёлый синтез одной темы не засоряет контекст остальных).

### Формат входного файла
Шапка задаёт общие настройки, ниже — по одной теме на строку:

```
регион: Москва
бренд: АртСтрой        # опционально, общий бренд для всех тем

статья  | как выбрать компанию для ремонта загородного дома под ключ
услуга  | сколько стоит дизайн-проект квартиры
рейтинг | лучшие студии дизайна интерьера в Москве
```

### Шаг 0 — парсинг (главная сессия)
- Строки `регион:` / `бренд:`, комментарии `#` и пустые — это конфиг, не темы.
- Тема — строка `тип | промпт`. Бренд/регион можно переопределить на строке темы:
  `тип | промпт | бренд | регион`.
- Для каждой темы вычисли `slug` (транслит, пробелы → подчёркивания) и рабочую
  папку `out/batch/<slug>/`, чтобы `citation-map.json` тем не перезатирали друг друга.

### Шаг 1 — волны по 4 (главная сессия)
Раздели темы на группы по **4** и запусти до 4 субагентов одновременно
(несколько вызовов `Agent`, `subagent_type: general-purpose`, в одном сообщении).
Дождись волны → следующая. **Лимит параллелизма — 4** (бережём бюджет WebSearch).

### Шаг 2 — работа субагента (одна тема)
Шаблон промпта субагента (подставь `{тип}`, `{промпт}`, `{бренд}`, `{регион}`,
`{slug}`):

```
Ты пишешь ОДНО GEO-ТЗ под продвижение в нейросетях. Рабочая директория скилла:
c:\Users\Алекс\Desktop\claude\skills\geo-tz-generator

1. Прочитай SKILL.md этого скилла, а также references/fan-out-taxonomy.md и
   references/geo-methodology.md. Следуй Фазам 1–3.
2. Тема: тип «{тип}», промпт «{промпт}», бренд «{бренд}», регион «{регион}».
3. Фаза 1: разложи промпт на веер (6 типов), построй таблицу подзапросов.
4. Фаза 2: через WebSearch прогони промпт + топ-подзапросы, сохрани фактуру в
   ./out/batch/{slug}/citation-map.json по схеме из SKILL.md.
5. Прочитай свой citation-map.json и выполни Фазу 3 (синтез ТЗ), ссылаясь на
   фактуру именно из этого файла.
6. Сохрани готовое ТЗ в C:\Users\Алекс\Desktop\тексты\Новые ТЗ\ по правилам
   именования (ТЗ_GEO_<Название>_<Регион>.md, при конфликте — с датой).
7. Верни КОРОТКИЙ отчёт: путь к ТЗ; число подзапросов в веере; число
   Mandatory-чанков; сколько источников собрано в citation-map; были ли проблемы
   с WebSearch. Без дампа citation-map.json.
```

При ошибке по своей теме субагент сообщает об этом в отчёте — остальные темы не
страдают.

### Шаг 3 — сводка (главная сессия)
Собери отчёты в таблицу:

| Тема | Тип | Файл ТЗ | Подзапросов | Mandatory-чанков | Источников | Статус |
|------|-----|---------|-------------|------------------|-----------|--------|

Отметь упавшие темы и предложи перезапустить.
