# SEO ТЗ Генератор v2 (анализ реального ТОПа)

Навык создаёт глубокие ТЗ для копирайтеров. Ключевое отличие v2: ТОП **реально
скачивается и парсится**, а не домысливается из сниппетов. Пайплайн из трёх фаз:

```
Фаза 1  Harvest    XMLRiver → 10 URL выдачи РФ (фильтр агрегаторов)   [скрипт]
Фаза 2  Aggregate  fetch + parseHtmlPage → top-map.json               [скрипт]
Фаза 3  Synthesize реверс-инжиниринг интента + сборка ТЗ              [модель]
```

Жёсткая часть (Фазы 1–2) — детерминированные скрипты: цифры в ТЗ защищаемы
(«так у 8 из 10 в ТОПе»). Умная часть (Фаза 3) — семантический анализ моделью
поверх реальных данных.

**Два режима работы:**
- **Одиночный** — пользователь даёт одну тему строкой `тип | запрос | регион`.
  Проходишь Фазы 1–3 сам и сохраняешь одно ТЗ. Описан ниже по фазам.
- **Батч** — пользователь даёт файл со списком тем. Каждое ТЗ пишет отдельный
  субагент в изолированном контексте, параллельно до 4. См. раздел
  [«Батч-режим»](#батч-режим-файл-со-списком-тем).

## Предусловия (один раз)

1. Соседний скил `google-yandex-seo-skills` должен быть на месте, и в нём выполнен
   `npm install` (даёт `cheerio` для парсера). Скрипт `analyze-top.js` сам проверит
   и подскажет, если чего-то нет.
2. Скопировать `config.example.json` → `config.json` и вписать `XMLRIVER_USER` /
   `XMLRIVER_KEY` (с https://xmlriver.com/queries/). `config.json` в `.gitignore`.

Если ключей XMLRiver нет — переходи в **ручной режим** (см. ниже): пользователь
даёт URL ТОПа, Фаза 1 пропускается.

## Батч-режим (файл со списком тем)

**Триггер:** пользователь скинул файл (`.md`/`.txt`) со списком тем или попросил
сделать ТЗ «пачкой / по списку / из файла». Тогда вместо одиночного прогона ты
становишься **оркестратором**: парсишь файл и раздаёшь темы субагентам, по одному
ТЗ на субагента, в изолированных контекстах. Так тяжёлый синтез Фазы 3 одной темы
не засоряет контекст остальных.

### Формат входного файла

Шапка задаёт общие настройки, ниже — по одной теме на строку `тип | запрос`:

```
регион: Москва
движок: yandex        # опционально, по умолчанию yandex

услуга  | ремонт квартир под ключ
статья  | авторский надзор в ремонте загородного дома
рейтинг | топ студий дизайна интерьера
```

Образец — [topics.example.md](topics.example.md).

### Шаг 0 — парсинг (главная сессия)

Прочитай файл и раздели на конфиг и темы:
- Строки `регион:` / `движок:`, комментарии `#` и пустые — это конфиг, не темы.
- Тема — строка `тип | запрос` (тип: услуга/статья/рейтинг/категория/обзор…).
- Регион можно переопределить на строке темы третьим полем: `тип | запрос | СПб`.
  Если не указан — общий из шапки; если и там нет — Москва.
- Для каждой темы вычисли `slug` (транслит, пробелы → подчёркивания) и рабочую
  папку `out/batch/<slug>/` — у каждой темы своя, чтобы `serp.json`/`top-map.json`
  не перезатирали друг друга.

### Шаг 1 — волны по 4 (главная сессия)

Раздели темы на группы по **4**. Для каждой группы запусти **до 4 субагентов
одновременно** — несколько вызовов `Agent` (`subagent_type: general-purpose`) в
одном сообщении идут параллельно. Дождись завершения волны → запусти следующую.
**Лимит параллелизма — 4** (XMLRiver и парсинг ТОПа), не больше.

### Шаг 2 — работа субагента (одна тема)

Каждый субагент получает ОДНУ тему и проходит **весь пайплайн Фаз 1–3** для неё,
читая инструкции из этого же `SKILL.md`. Промежуточные артефакты пишет в свою
`out/batch/<slug>/`, финальное ТЗ — в `C:\Users\Алекс\Desktop\тексты\Новые ТЗ\`.
Возвращает короткий отчёт (НЕ дамп `top-map.json`).

Шаблон промпта субагента (подставь `{тип}`, `{запрос}`, `{регион}`, `{движок}`,
`{slug}`):

```
Ты пишешь ОДНО SEO-ТЗ. Рабочая директория скилла:
c:\Users\Алекс\Desktop\claude\skills\seo-tz-generator

1. Прочитай SKILL.md этого скилла и следуй Фазам 1–3.
2. Тема: тип «{тип}», запрос «{запрос}», регион «{регион}», движок {движок}.
3. Фаза 1: node scripts/harvest.js --query "{запрос}" --engine {движок} \
   --region "{регион}" --out ./out/batch/{slug}/serp.json
4. Фаза 2: node scripts/analyze-top.js --in ./out/batch/{slug}/serp.json \
   --out ./out/batch/{slug}/top-map.json
5. Прочитай свой out/batch/{slug}/top-map.json и выполни Фазу 3 (синтез ТЗ),
   ссылаясь на цифры именно из этого файла.
6. Сохрани готовое ТЗ в C:\Users\Алекс\Desktop\тексты\Новые ТЗ\ по правилам
   именования из SKILL.md (ТЗ_<Название>_<Регион>.md, при конфликте — с датой).
7. Верни КОРОТКИЙ отчёт: путь к ТЗ; объём (зн.); число Mandatory-H2; были ли
   проблемы с ТОПом (сколько страниц спарсилось из 10). Без дампа top-map.json.
```

При ошибке по своей теме (нет ключей, XMLRiver вернул ошибку, ТОП не спарсился)
субагент сообщает об этом в отчёте — остальные темы не страдают.

### Шаг 3 — сводка (главная сессия)

Собери отчёты всех субагентов в таблицу и покажи пользователю:

| Тема | Тип | Файл ТЗ | Объём | Mandatory-H2 | Статус |
|------|-----|---------|-------|--------------|--------|

Отметь темы, упавшие с ошибкой, и предложи перезапустить их.

## Фаза 1 — Harvest (сбор реального ТОПа)

### Формат ввода
Пользователь задаёт запрос строкой:

```
тип | запрос | регион
```

- **тип** — тип страницы (статья/услуга/рейтинг/категория/обзор…). `рейтинг` —
  листинг/сравнительная страница (ТОП-N): в ТЗ закладывается сетка критериев и
  сравнение участников. Это всё равно **ТЗ копирайтеру**, а не сама статья —
  для написания готового рейтинга есть отдельный скилл `rating-writer`.
- **запрос** — поисковый запрос.
- **регион** — необязателен. Город (`Санкт-Петербург`) ИЛИ числовой код `lr`.
  **Если не указан — Москва.**

Примеры:
- `статья | авторский надзор в ремонте загородного дома` → регион Москва (по умолчанию)
- `услуга | ремонт квартир под ключ | Санкт-Петербург`
- `категория | кухни на заказ | 54` (Екатеринбург по коду lr)

Движок по умолчанию — Яндекс. Регион-город работает только для Яндекса; для Google
XMLRiver собирает выдачу по России (скрипт честно об этом предупредит).

### Запуск
Передай регион как есть — название города или код. Скрипт сам резолвит название в
код `lr` через [references/regions.json](references/regions.json).

```bash
cd skills/seo-tz-generator
# регион по умолчанию (Москва):
node scripts/harvest.js --query "<запрос>" --engine yandex --top 10 --out ./out/serp.json
# с регионом (город или код lr):
node scripts/harvest.js --query "<запрос>" --engine yandex --region "Санкт-Петербург" --out ./out/serp.json
```

Скрипт получает выдачу, отбрасывает агрегаторы (`references/aggregators.txt`) и
дубли доменов, сохраняет `out/serp.json` с 10 прямыми конкурентами.

**Ручной режим (без XMLRiver):** пропусти harvest, собери список URL от
пользователя и передай их сразу в Фазу 2 через `--urls`.

## Фаза 2 — Aggregate (карта рынка)

```bash
node scripts/analyze-top.js --in ./out/serp.json --out ./out/top-map.json
# или вручную:
node scripts/analyze-top.js --urls "https://a.ru/page,https://b.ru/page" --query "<запрос>" --out ./out/top-map.json
```

Скачивает каждую страницу, разбирает переиспользуемым `parseHtmlPage` и строит
`top-map.json`:

- `volume` — медиана объёма (слова/символы), p25/p75, рекомендуемый диапазон (медиана +10–15%).
- `headingMap[]` — тематические блоки H2/H3 с `count`, `share`, `weight`
  (**Mandatory ≥70% / Recommended 40–70% / Optional <40%**) и примерами заголовков.
- `tech` — медианы списков/таблиц, доля FAQ, объединение типов schema, доля
  «ответ в первом абзаце» (`definitionIntroShare`).
- `lexicon` — частотные униграммы и биграммы из заголовков + первых абзацев (реальные LSI).
- `competitors[]` — поплощадочные данные (объём, H1/H2/H3, schema).

**Прочитай `out/top-map.json` перед Фазой 3.** Это фактура, на которой строится ТЗ.

## Фаза 3 — Synthesize (анализ интента + сборка ТЗ)

Сначала выполни поиск в Google/Яндекс по запросу для контекста, затем — анализ ниже.
Везде, где даёшь цифру или вес, **ссылайся на `top-map.json`** (например,
«объём 9 200 зн. — медиана ТОПа +12%», «H2 “Стоимость” — Mandatory, 8/10»).

### Шаг 1: Моделирование интерпретации запроса поисковым ИИ (реверс-инжиниринг)

Анализируемый запрос: `[ТИП СТРАНИЦЫ] + [ЗАПРОС] + пояснение`.

**1.1. Распознавание сущностей (NER) и атрибутов:** обнаруженные сущности (тип,
нормальная форма); явные и неявные атрибуты.
**1.2. Контекстуализация через онтологию/таксономию:** активированная
таксономическая ветка (основная + вышестоящие категории); ключевые узлы онтологии
(связанные сущности, отношения, атрибуты).
**1.3. Интент и фокус внимания:** интенты **списком по приоритету** (от главного к
косвенному); распределение Salience (ключевые сущности с наибольшим весом);
семантические связи в запросе.
**1.4. Внутреннее представление и ожидания ИИ:** расширенная/нормализованная форма
запроса; семантический чек-лист ожиданий от контента; ключевые сигналы
релевантности; обработка неоднозначности; концептуальная оценка уверенности.

> На этом шаге сверяйся с `top-map.json`: ожидания ИИ должны пересекаться с
> `headingMap` (что реально раскрывает ТОП) и `lexicon` (какими словами).

### Шаг 2: Прикладные рекомендации по странице

**2.1. Содержание:** какие сущности/атрибуты/онтологические связи (из запроса,
онтологии и `lexicon`) раскрыть; как отразить контекст.
**2.2. Структура (таксономия + интент):** иерархический план H1–H3, где **каждый
H2 помечен весом из `headingMap`** (Mandatory/Recommended/Optional, X/N конкурентов).
Обязательные разделы; структурные элементы (таблицы/списки/FAQ/схемы — по `tech`).
Для типа **рейтинг** структура строится вокруг сетки критериев и сравнительной
таблицы участников (ТОП-N), карточек позиций и итогового выбора — заложи это в план H2.
**2.3. Тип контента:** формат страницы; требования к «Первому экрану» — сформулируй
ключевые выводы статьи, отвечающие на главный интент (учитывай `definitionIntroShare`).
**2.4. Ключевые слова:** интегрируй термины из `lexicon` (с частотностью) + сущности из Шага 1.
**2.5. Взаимосвязь с другими сущностями структуры сайта** (внутренняя перелинковка).

### Gap-анализ (точки роста)

Сопоставь свою «нейроструктуру идеальной страницы» с `headingMap`. Темы, которых
**нет ни у кого** в ТОПе, вынеси отдельным блоком **Gap** — это применяется как
требование Content Effort (см. `references/methodology.md`, п.1).

### Требования к контенту

Внедри все 4 правила из [references/methodology.md](references/methodology.md):
Content Effort, декларативные заголовки, Original Media, сигнальная когерентность.

## Формат итогового ТЗ (шаблон)

1. **Метаданные:** Title, Description (с опорой на интент и `lexicon`).
2. **Технические требования:**
   - Объём: `volume.recommendedMinChars–MaxChars` (медиана ТОПа +10–15%).
   - Списков: мин. `tech.listMedian`; таблиц: мин. `tech.tableMedian`.
   - FAQ: если `tech.faqShare ≥ 0.4` — обязателен; schema: `tech.schemaTypesUnion`.
3. **Структура (план):** иерархия H1→H2→H3, у каждого H2 — вес и обоснование (X/N).
4. **Gap-блоки:** чего нет у ТОПа → обязательное уникальное усилие.
5. **Ключевые слова:** основные (из `lexicon.unigrams`), фразы (`lexicon.bigrams`),
   тематические; правила вхождения (частотность, прямое/разбавленное).
6. **Фактура:** найденные уникальные цифры/факты/ссылки.

### Куда сохранять готовое ТЗ

**Папка по умолчанию:** `C:\Users\Алекс\Desktop\тексты\Новые ТЗ\`
Сохраняй итоговое ТЗ туда (Markdown), если пользователь явно не указал другой путь.
Имя файла — `ТЗ_<Короткое_название>_<Регион>.md` (пробелы → подчёркивания), например
`ТЗ_Консультация_стоматолога-ортопеда_Ростов.md`. Если файл с таким именем уже есть —
добавь дату `_ГГГГ-ММ-ДД`, чтобы не перезаписать чужое ТЗ.

Промежуточные артефакты пайплайна (`serp.json`, `top-map.json`) остаются в `out/` —
в папку с готовыми ТЗ кладётся **только финальный Markdown**.
