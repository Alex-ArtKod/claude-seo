# SEO ТЗ Generator — автономный многопоточный пайплайн

## Что делает скилл

Берёт **один файл со списком тем** (по одной теме на строку), и для каждой темы
пишет готовое **ТЗ** в отдельный `.md` файл. Каждое ТЗ строится по промпту-стратегу
(`references/tz-strategist-prompt.md`): реверс-инжиниринг того, как поисковый ИИ
интерпретирует запрос (сущности, онтология, таксономия, интент, внимание) + прикладные
рекомендации по содержанию и структуре страницы. Анализ опирается на **живую выдачу
Google и Яндекса**, полученную через **xmlriver** (`scripts/xmlriver_serp.py`): обе
выдачи фетчатся для каждой темы и анализируются вместе (консенсус + различия движков).

Работает **без подтверждений**, **в параллель**, и **переживает обрыв сессии и лимиты**
(всё состояние — на диске; см. State model и Limit handling). Выход совместим по духу
с `seo-writer`: получившиеся ТЗ можно затем скормить ему для написания статей.

---

## Autonomy contract

- **Без вопросов.** Все разрешения выданы. Никогда не останавливайся, чтобы спросить.
- **Выдачу не выдумывай.** Источник правды по выдаче — JSON от `xmlriver_serp.py`.
  Если SERP пуст/недоступен — отметь это в ТЗ и работай от семантики запроса.
- **Возобновляемость с диска.** Готовое ТЗ = существующий выходной `.md`. Убитая или
  залимиченная сессия возобновляется простым повторным сканом списка.

---

## Вход: файл со списком тем

Один текстовый файл (`.md` / `.txt`). Формат строки:

```
<тема/запрос> [| <тип>] [| <ключи через запятую>]
```

Шапка задаёт значения по умолчанию для всех строк (любые можно опустить):

```
# region: Ростов-на-Дону
# type: статья
# loc: 1011969          (опц. — точный гео-id xmlriver; иначе регион клеится к запросу)
# output: ./ТЗ          (опц. — куда писать; по умолчанию папка ./ТЗ рядом со списком)
# sites: example.ru, example2.ru   (опц. — приоритетные сайты из ТОПа)

имплантация зубов под ключ | услуга | имплант под ключ, цена импланта
сколько стоит имплантация | статья
протезирование зубов            (берёт type из шапки)
```

- Строки, пустые или начинающиеся с `#` (кроме директив шапки), игнорируются.
- `type` из строки переопределяет `type` из шапки. Если типа нет нигде — определи его
  по интенту запроса и зафиксируй в ТЗ.
- Если пользователь задал регион/тип в реплике («у всех тип статья, регион Ростов») —
  это перекрывает шапку файла для всего прогона.

**Как найти файл:** если путь не назван явно — найди в текущей папке единственный
подходящий список (`.md`/`.txt`, не являющийся уже сгенерированным ТЗ). Несколько
кандидатов или ноль → перечисли найденное и попроси указать файл (это единственное
исключение из «без вопросов» — без входа работать нельзя).

---

## Выход

Один файл на тему: `{output}/{slug}.md`, где `{slug}` — тема, очищенная под имя файла
Windows (убрать `/ \ : * ? " < > |`, обрезать пробелы; длину — до ~120 символов). Если
заданы ключи — допиши краткий заголовок темы в начало файла. Папка вывода — из директивы
`# output:` либо `./ТЗ` рядом со списком.

Файл ТЗ содержит ровно вывод промпта-стратега (Шаг 1 + Шаг 2 + иерархическая структура
страницы) на русском.

---

## State model (как работает возобновление)

| Сигнал | Значение |
|--------|----------|
| `{output}/{slug}.md` существует и непустой | Готово. Пропустить. |
| `{output}/.{slug}.inprogress` свежий (<6ч) | Взято в работу. Не назначать. |
| маркер старше 6ч | Протух — переназначить. |
| ничего из выше | Свежая тема. Делать с нуля. |

---

## Оркестратор (основной поток скилла)

`N` = целочисленный аргумент (по умолчанию 1) = **сколько ТЗ сделать за этот прогон**
(и максимальная параллельность). Оркестратор останавливается, как только N тем готово,
даже если в списке осталось больше.

1. **Прочитай** файл списка и шапку. Построй очередь тем (исключи готовые и свежие
   `.inprogress`). Создай папку вывода, если нет.
2. Очередь пуста → выведи финальное summary и стоп.
3. **Раздай до N агентов**, по одной теме на агента, через Agent tool с
   `run_in_background: true`, `subagent_type: general-purpose`. Перед раздачей запиши
   `{output}/.{slug}.inprogress` с UTC-временем. Дай каждому агенту: тему, тип, регион,
   loc (если есть), список приоритетных сайтов, абсолютные пути к этому скиллу
   (`scripts/xmlriver_serp.py`, `references/tz-strategist-prompt.md`) и папке вывода, и
   инструкцию: **«Выполни Per-topic pipeline из seo-tz-generator/SKILL.md для этой темы
   от начала до конца. Вопросов не задавай.»** Считай `total_dispatched`; не раздавай
   больше N тем суммарно.
4. **Держи до N в работе, но не больше N всего.** Когда агент закончил (его `.md`
   появился) и `total_dispatched < N` — бери следующую тему. Как только
   `total_dispatched == N`, дождись всех в работе и иди к шагу 6.
5. **При лимитах** → см. Limit handling.
6. N тем готово → выведи: `✓ Готово: {N} ТЗ в {output}/.`

Для `N = 1` выполни Per-topic pipeline инлайн (без субагента) ровно для одной темы и стоп.

## Per-topic pipeline

Каждый агент (или инлайн-прогон) делает это для ОДНОЙ темы. Точку входа определи по
State model.

### Шаг 0 — Claim
Создай `{output}/.{slug}.inprogress` (UTC-время), если его нет. Загрузи один раз
`references/tz-strategist-prompt.md`.

### Шаг 1 — SERP через xmlriver (Google + Яндекс)
Сформируй запрос из темы (тип помогает осмыслить интент, в строку запроса его обычно
не клеят). Получи **обе** выдачи — Google и Яндекс — двумя вызовами:

```
python "{skill}/scripts/xmlriver_serp.py" "<тема>" --region "<регион>" --num 10 --engine google
python "{skill}/scripts/xmlriver_serp.py" "<тема>" --region "<регион>" --num 10 --engine yandex
```

(если в шапке задан `loc` — добавь `--loc <id>` для **Google**; для Яндекса гео берётся
из региона, вклеенного в текст запроса, — Google-овский `loc` ему не передаётся). Скрипт
берёт ключи из `~/.config/claude-seo/xmlriver.json` или env `XMLRIVER_USER`/`XMLRIVER_KEY`
и печатает JSON (позиция, url, домен, заголовок, сниппет).

Обрабатывай движки независимо — пустая или недоступная выдача одного не отменяет другой:
- Exit 4 / пустой `results` у движка → пометь этот движок как «без данных» и работай по
  второму; если пусты оба — переходи к Шагу 3 от семантики запроса.
- Exit 2 (нет ключей) → залогируй в `{output}/.tz-log.md` и в summary; ТЗ собери от
  семантики, пометив «без данных выдачи». Не падай молча.
- Exit 3 (ошибка API/сети) → один повтор для этого движка; затем как пустая выдача + лог.

### Шаг 2 — (опц.) Структура конкурентов
Если нужна глубина по структуре — дофетчи топ-3 страницы выдачи через общий
`scripts/fetch_page.py` основного скилла seo (если доступен) и извлеки их H1–H3. Не
обязательно: для большинства ТЗ хватает заголовков и сниппетов из SERP-JSON.

### Шаг 3 — Применение промпта-стратега
Подставь в `tz-strategist-prompt.md`: `{{PAGE_TYPE}}`, `{{QUERY}}`, `{{REGION}}`,
`{{SERP_JSON_GOOGLE}}` и `{{SERP_JSON_YANDEX}}` (выводы Шага 1 по каждому движку),
`{{TOP_SITES}}` (директива `# sites:`). Выполни промпт полностью: Шаг 1
(реверс-инжиниринг) + Шаг 2 (рекомендации) + иерархическая структура страницы
(H1/H2/H3…). Только русский; обе выдачи анализируй, не выдумывай.

### Шаг 4 — Сохранение
Сохрани результат в `{output}/{slug}.md`. Удали `.{slug}.inprogress`. Выведи:
`✓ ТЗ готово: {slug}.md`. Верни управление оркестратору.

## Limit handling (автономное возобновление)

При лимите использования:
1. Оставь файловую систему возобновляемой (частичная работа + `.inprogress`).
2. Через скилл `/schedule` создай cron-рутину, перезапускающую `/seo-tz-generator {N}`
   на этом списке **каждые 2 часа**.
3. На каждом пробуждении пересканируй список. Лимиты ещё не сняты → тихо выйди до
   следующего тика. Сняты → продолжи с диска.
4. Когда очередь полностью обработана — **удали cron-рутину** через `/schedule`.

## Настройка xmlriver (один раз)

Создай `~/.config/claude-seo/xmlriver.json`:

```json
{ "user": "ВАШ_USER_ID", "key": "ВАШ_KEY" }
```

(или экспортируй `XMLRIVER_USER` и `XMLRIVER_KEY`). User id и key — в личном кабинете
xmlriver.com.

**Про регион и `loc`:** для Google-выдачи xmlriver `loc` (Google geotarget id)
обязателен. По умолчанию скрипт ставит `loc=2643` (Россия) и добавляет регион к тексту
запроса (напр. «имплантация зубов под ключ Ростов-на-Дону») — проверено, для городов РФ
это даёт корректно локализованную выдачу. Если нужен точный городской geotarget — задай
его в шапке списка через `# loc: <id>` (например, Москва = `1011969`); тогда регион в
запрос не подклеивается.

## Error handling

| Сценарий | Действие |
|----------|----------|
| Файл списка не найден / неоднозначен | Перечисли кандидатов, попроси указать (единственный вопрос) |
| Нет ключей xmlriver (exit 2) | Залогируй, собери ТЗ от семантики с пометкой «без данных выдачи» |
| Ошибка API/сети xmlriver (exit 3) | 1 повтор, затем как пустая выдача + лог |
| Пустая выдача (exit 4) | ТЗ от семантики запроса, пометить в ТЗ |
| Тема даёт пустой/битый slug | Подставь `tz-{индекс}` как имя файла, залогируй |
| Файл ТЗ уже существует | Пропусти (готово), если непустой |
| Лимит использования | Пауза, cron каждые 2ч через /schedule, продолжить на пробуждении |
| Протухший `.inprogress` (>6ч) | Переназначь тему |
